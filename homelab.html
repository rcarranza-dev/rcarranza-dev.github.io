The Stack

Every component chosen for a reason. Every port, every binding, every network boundary is a deliberate security decision.

HTTPS termination with Let's Encrypt TLS 1.2/1.3. HTTP auto-redirects to HTTPS. LAN-only access via UFW — no ports forwarded to the internet.

security hardening

Defense in Depth

Every service is evaluated by how many independent security layers protect it. The goal is never a single point of failure.

Service Protection Layers Depth
SSH (Port 22) Binding + UFW + Key Auth + Fail2Ban 4 layers
HTTP/HTTPS (80/443) Binding + UFW + Nginx Proxy 3 layers
Open WebUI (3000) Localhost Bind + Nginx 2 layers
Ollama (11434) UFW Only (was localhost + UFW) 1 layer ⚠

Additional hardening: AppArmor (MAC enforcement), Auditd (file integrity monitoring on /etc/passwd, /etc/sudoers, sshd_config), unattended security patches, root account locked, lid close and sleep targets disabled for server uptime.

threat model

AI-Specific Attack Vectors

This lab isn't just running an LLM — it's designed to understand and defend against the attack surface that comes with it.

Prompt Injection via Email
Hidden instructions in email body — white text, HTML comments, Unicode tricks — designed to manipulate LLM behavior when processing emails.
Metadata-only processing. The AI never reads email body content.
BEC-via-LLM
Attacker crafts email content designed to make an AI misclassify a malicious request as legitimate, bypassing traditional BEC detection.
Human-in-the-loop approval for all write actions. No autonomous execution.
Steganography in Attachments
Instructions hidden in image pixels or document metadata, invisible to the user but potentially parsed by an AI processing the file.
Attachment processing restricted to metadata extraction only.
QR Code Injection
Encoded URLs or instructions embedded in images within emails, designed to be decoded and followed by AI vision capabilities.
No image content analysis on email attachments. Vision features disabled for email pipeline.
design decisions

Why This Way?

Every architecture choice has a security rationale. Here are the ones that matter most.

Why local instead of cloud APIs?
Zero vendor dependency. No token costs, no policy changes breaking agents overnight, no data leaving the machine. A friend's Claude-powered Discord bot lost its persona when Anthropic changed API behavior. Another friend's agentic cron job ran up a massive bill from an infinite retry loop. We own the entire stack — the model, the data, and the infrastructure.
Why metadata-only email processing?
The email body is the single largest prompt injection attack surface. By restricting the AI to metadata — sender domain, date, has-attachment, subject line — we eliminate the entire category of attacks that rely on hidden content in email bodies. The tradeoff is functionality, but the security gain is worth it.
Why human-in-the-loop for all write actions?
An AI that can read your email, check your calendar, and take actions autonomously is one prompt injection away from sending emails on your behalf. Every write action — calendar events, task creation, email responses — requires explicit human approval before execution.
Why Ollama on 0.0.0.0 instead of localhost?
Docker containers can't reach localhost on the host machine without --network host mode (which sacrifices Docker network isolation). Binding Ollama to all interfaces and restricting access via UFW to only the Docker subnet was the best tradeoff — but it reduced defense from 2 layers to 1. This is a known risk, documented and tracked.
